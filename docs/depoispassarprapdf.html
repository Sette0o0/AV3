1. Introdução

Este documento reúne o entendimento do produto, requisitos e orientações de informação e navegação para a construção da primeira interface gráfica (GUI) do sistema de gestão da produção de aeronaves da Aerocode. O objetivo é transformar a aplicação atual em linha de comando (CLI) em uma SPA (Single Page Application) em React, priorizando usabilidade, segurança e escalabilidade para clientes corporativos.

2. Objetivos do Projeto

  • Modernizar a interface do sistema (migração CLI → GUI).
  • Reduzir a curva de aprendizado de novos usuários.
  • Permitir gerenciamento claro de funcionários, aeronaves, peças, etapas e testes.
  • Fornecer controles de permissão por perfil (Administrador, Engenheiro, Operador).
  • Preparar o produto para adoção por grandes fabricantes aeronáuticas.

3. Público‑alvo

A Empresa

Grandes fabricantes de aeronaves e empresas do setor aeroespacial que gerenciam múltiplos projetos, equipes grandes e processos de produção complexos (ex.: Embraer, Boeing, Airbus, Bombardier, Lockheed Martin).

Usuários Principais

  • Administrador (Adm): responsável por gerir contas e permissões; acesso total ao sistema.
  • Engenheiro: engenheiro de produção / aeronáutico — gerencia aeronaves, peças, etapas e testes.
  • Operador: usuários de chão de fábrica — atualizam o status de peças e etapas; permissões restritas.

4. Requisitos

4.1 Requisitos Funcionais

  1. Login com verificação de credenciais e sessão (perfis: Adm, Engenheiro, Operador).
  2. CRUD de funcionários (Adm).
  3. CRUD de aeronaves, peças, etapas e testes.
  4. Alteração de status de peças e etapas.

4.2 Requisitos Não‑Funcionais

  • Aplicação como SPA em React (protótipo navegável).
  • Segurança: controle de sessão, segregação de perfil e proteção de dados sensíveis.
  • Compatibilidade com Windows 10+ e Ubuntu 24.04+.

5. Hierarquia de Informações

A hierarquia de informações determina o que aparece primeiro e com maior destaque na interface. Abaixo segue a proposta, do mais importante ao menos destacado:

  1. Identificação do sistema / usuário (nome da empresa, Menu de navegação, Sair) — topo.
  2. Área de Conteúdo — lista / detalhe / formulário.
  3. Controles primários (Criar, Detalhes) — próximos ao conteúdo principal.
  4. Filtros e Pesquisa — colocados acima das listas para fácil acesso.
  5. Contexto — para orientar a posição dentro da hierarquia (ex.: Home / Aeronaves / A320-001).
  6. Informações secundárias (rodapé).

Observação sobre permissões: elementos e ações devem aparecer conforme o perfil do usuário. Por exemplo, o botão "Cadastrar Aeronave" só será visível para o Adm e o Engenheiro.

6. Fluxo de Navegação (User Flow)

Fluxo básico dos usuários ao acessar o sistema:

  1. Login → validação de credenciais
  2. Home / Inicio
    1. Seleção da seção (Aeronaves)
      • Cadastrar aeronave
    2. Lista de aeronaves → seleção de aeronave para ver Detalhes
    3. lista de peças, etapas e testes → ações possíveis (Adicionar, Avançar Etapa, Atualizar Status)

    1. Seleção da seção (Funcionários)
      • Cadastrar funcionario
    2. Lista de funcionários → seleção de funcionário para ver Detalhes
  3. Logout

8. Conclusão

O protótipo e o documento colocam as bases para a transformação do sistema CLI em uma aplicação web moderna. O foco nas permissões por perfil, na hierarquia de informação e nos fluxos de usuário garantirá que o produto atenda tanto às necessidades operacionais do chão de fábrica quanto às demandas de gestão e auditoria das equipes de engenharia e administração.
